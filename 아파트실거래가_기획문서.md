# 엑셀아파트(가칭) 기획 문서

## 1. 한줄 컨셉

회사에서 "엑셀 보는 척" 하면서 사실은 관심 아파트 단지의 실거래가를 확인하는 사이트.
엑셀코스피(excelkospi)의 위장 UI 컨셉을 차용하되, 콘텐츠는 주식 시세 대신 국토교통부 아파트 실거래가로 대체한다.

## 2. 타겟 사용자

- 내 집 마련을 준비 중이라 시세를 자주 확인하고 싶은 직장인
- 관심 단지 여러 개를 등록해두고 거래 발생 여부를 모니터링하고 싶은 사람
- "회사에서 부동산 시세 보는 거 티 안 나게" 보고 싶은 사람

## 3. 주식 버전과의 결정적 차이 (기획 시 유의점)

| 구분 | 주식(엑셀코스피) | 아파트(이 사이트) |
| --- | --- | --- |
| 갱신 주기 | 초/분 단위 실시간 | 거래 신고 기준, 통상 D+1~30일 지연 |
| 데이터 성격 | 호가/체결가 | 실제 신고된 거래 내역 (호가 아님) |
| 재미 포인트 | 실시간 등락의 긴장감 | "신고 발견"의 발견형 재미, 관심단지 추적 |
| 자동 갱신 | 의미 있음 (시세 변동) | 의미 낮음 (하루 한두 번 배치 갱신으로 충분) |

→ "실시간 깜빡이는 시세"보다는 "오늘 새로 올라온 신고 내역", "내 단지 최근 거래"를 발견하는 경험에 무게를 둔다.

## 4. 시트 구성안 (단계적으로 확장)

### A시트 — 관심단지 (핵심 화면)
- 사용자가 등록한 단지의 최근 실거래가 목록
- 컬럼: 단지명 | 전용면적 | 거래일 | 거래금액 | 직전 거래 대비 등락 | 층
- 등록은 로컬스토리지 기반 (로그인 없이도 사용 가능, 엑셀코스피와 동일한 원칙)

### B시트 — 실거래 속보
- 최근 N일간 신고된 거래를 지역별로 나열하는 헤드라인 시트
- 예: "OO아파트 84㎡ 12억 5천 신고 (강남구, 6/29)"
- 뉴스 시트의 빈자리를 대체하는 콘텐츠형 화면

### 보류 (향후 검토)
- **C시트(지역 비교)**: 동/구 단위 평균 거래가, 거래량, 전월 대비 변동률 — 1차 범위 제외, 추후 추가
- **D시트(업무 위장용 시트)**: 엑셀코스피의 "월간 업무 진행 현황"과 동일한 역할, 보스 긴급키 등 — 1차 범위 제외, 추후 추가
- 모의투자에 대응하는 "갭투자 시뮬레이터" / "전세가율 계산기" 등도 1차 범위에서 제외하고 추후 확장 항목으로 둔다

## 5. 데이터 연동 설계

### 5.1 API
- 공공데이터포털 "국토교통부_아파트매매 실거래자료" API 사용 (서비스키 발급 완료, 활용기간 2026-06-30 ~ 2028-06-30)
- End Point: `https://apis.data.go.kr/1613000/RTMSDataSvcAptTrade/getRTMSDataSvcAptTrade`
- 요청 파라미터: `LAWD_CD`(법정동코드 앞 5자리, 필수), `DEAL_YMD`(계약년월 6자리, 필수), `serviceKey`(필수), `pageNo`/`numOfRows`(선택)
- 응답: XML, 데이터 갱신주기는 **일 1회** (기술문서 확인) → 기획대로 하루 1~2회 배치 갱신이면 충분
- 응답 핵심 항목: `aptNm`(단지명), `umdNm`(법정동), `jibun`(지번), `excluUseAr`(전용면적㎡), `dealYear/dealMonth/dealDay`(계약일), `dealAmount`(거래금액, 만원 단위), `floor`(층), `buildYear`(건축년도), `dealingGbn`(중개/직거래), `aptDong`(동명), `cdealType`(해제여부)
- **파싱 유의사항**
  - `dealAmount`는 숫자가 아니라 `"12,000"`처럼 쉼표가 포함된 문자열로 내려오므로 쉼표 제거 후 정수 변환 필요
  - `cdealType`에 값이 있으면 해당 거래는 해제(취소)된 건이므로, B시트(실거래 속보)에서 실제 거래로 오인하지 않도록 별도 표시하거나 제외 처리 필요
- 에러코드 체계 존재 (예: 20=활용승인 미완료, 22=일일 트래픽 초과, 30=잘못된 서비스키, 31=기간 만료) → 프로토타입 단계에서 기본적인 에러 핸들링 필요

#### 실제 테스트 호출 결과 (2026-06-30 확인)
- LAWD_CD=11680(강남구), DEAL_YMD=202605로 테스트 → 정상 응답(`resultCode=000`) 확인
- 강남구 한 달치만 `totalCount=443`건으로 데이터량 충분함을 확인
- `cdealType`이 빈 값이면 정상 거래, 값이 있으면 해제(취소) 거래라는 점을 실제 응답으로도 확인
- `aptDong`(동명)은 대부분 빈 칸으로 옴 (단지가 동 구분 없이 통합 표기되는 경우가 많음) → A/B시트 설계 시 이 컬럼에 과도하게 의존하지 않을 것
- 한글 인코딩, 한 달 분량 호출 모두 문제 없이 동작 → 프로토타입 진행에 무리 없음 확인됨

### 5.1-1 보조 데이터 — 세대수/년식 (정적 스냅샷 방식)
- 아파트매매 실거래자료 API에는 세대수 항목이 없음
- **1차 시도**: 공동주택관리정보시스템의 "공동주택 단지 목록 제공 서비스"(AptListService, 단지명→kaptCode 검색용)와 "공동주택 기본 정보제공 서비스"(AptBasisInfoServiceV4, `getAphusBassInfoV4`, 세대수·사용승인일 조회)를 실시간 연동 시도
  - 기본정보 API(V4, 응답 JSON)는 실제 키로 정상 동작 확인 (`kaptdaCnt`=세대수, `kaptUsedate`=사용승인일, 응답구조 `response.body.item` 단일 객체)
  - 하지만 단지 **목록** API는 해당 계정에 활용신청이 안 되어 있어(세대수 API와 별개 활용신청 필요) 아파트명 → kaptCode 매칭 경로가 없음 → 이 방식은 보류
- **최종 채택**: K-apt(공동주택관리정보시스템) 홈페이지가 로그인·인증 없이 제공하는 "전체 단지 기본정보" 엑셀 다운로드(`https://www.k-apt.go.kr/web/board/goKaptBasicExcelDownload.do`, 전국 21,641개 단지, 매주 금요일 갱신)를 활용
  - 2026-07-03 스냅샷에서 서울 25개구 3,174개 단지만 추출해 `seoul_apt_complex.json`(`kaptCode`, `kaptName`, `gu`, `dong`, `buildYear`, `householdCount`)으로 저장소에 정적 내장 (`seoul_lawd_codes.json`과 동일한 패턴)
  - 실거래가 API의 `aptNm`(+`umdNm`)과 이 파일의 `kaptName`(+`dong`)을 공백/괄호 제거 후 정규화하여 매칭 (완전일치 중 동 일치 우선 → 완전일치 → 부분포함 순)
  - 장점: 추가 API 활용신청·서비스키·런타임 호출이 전혀 필요 없음. 단점: 스냅샷이므로 신축 단지 반영 지연, 표기 차이가 큰 단지는 매칭 실패 가능(→ "-" 표시)
- 년식은 위 정적 데이터 대신 이미 보유한 실거래가 API의 `buildYear`(준공년도)를 그대로 사용 (매칭 실패 여부와 무관하게 항상 표시됨)

### 5.3 서버 구조 (CORS 및 키 보호 목적)
- 브라우저에서 직접 호출 불가 (CORS, serviceKey 노출 문제) → 서버 프록시 필수
- Cloudflare Pages Functions로 프록시 + 캐시 레이어 구성
- 거래 데이터는 하루 1~2회 배치로 갱신해 KV 또는 D1에 저장 → 응답 속도 확보 및 API 호출량 절감
- 캐시 무효화 기준: 매일 새벽 1회 전체 갱신 + 사용자 요청 시 캐시 우선 응답

## 6. 위장 모드

- 1차 범위는 "엑셀(시트)" 위장만으로 한정
- 메신저(Outlook 느낌) 위장 등은 추후 단계에서 검토
- 보스 긴급키, 위장 모드 토글 등은 엑셀코스피의 패턴을 그대로 차용

## 7. 단계적 로드맵

1. **기획 문서** (현재 단계)
2. **프로토타입**: 서울 일부 구 한정, 법정동코드 매핑 + 실거래가 API 연동 + 엑셀 UI 흉내낸 단일 페이지
3. **기능 확장**: 관심단지 등록(로컬스토리지), 지역 비교 시트, 위장 모드, 전국 확장, 알림 기능

## 8. 미정/확인 필요 사항

- 전국 단위 법정동코드 매핑 작업 범위와 시점
- 거래 발생 알림 방식 (텔레그램 연동 여부 등 엑선코스피처럼 외부 채널을 쓸지)
- 배포 환경 확정 (Cloudflare Pages 기준으로 진행할지 여부)
